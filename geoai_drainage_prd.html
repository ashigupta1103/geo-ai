Section 1 — Problem
Why rural India floods — and what's missing
No accurate terrain model
Manual surveys miss fine-scale elevation variation in abadi zones. Slope estimates are wrong, drainage is placed incorrectly.
Natural flow paths ignored
Roads are elevated without planned outflow. Water stagnates in low-lying plots, damaging kutcha homes and crops.
No flood prediction layer
Disaster response is entirely reactive. There is no model identifying which parcels will waterlog before monsoon.
Drainage built without optimization
Government schemes lack terrain intelligence, leading to inefficient routing, excess excavation cost, and low impact.
One-line problem
Raw SVAMITVA drone point clouds exist, but there is no automated, intelligent pipeline that converts them into actionable, cost-optimized drainage infrastructure for village planners.
Section 2 — Goals & Non-Goals
What this system must do
Goals
  • G1 — Automatically classify drone point cloud into ground, building, vegetation, road, water
  • G2 — Generate survey-quality Digital Terrain Model (DTM) at ≤1m resolution
  • G3 — Detect natural drainage paths, micro-watersheds, and depressions
  • G4 — Predict waterlogging hotspots using ML on terrain + hydrological features
  • G5 — Generate optimized drainage network (minimizing cost, excavation, disruption)
  • G6 — Export GIS-ready layers (GeoTIFF, SHP) and host interactive dashboard
Non-Goals
  • — Real-time flood monitoring (sensor integration is out of scope)
  • — Urban-scale processing (village scale only at this stage)
  • — Actual construction management or contractor interfacing

M1
Ingestion & Preprocessing
Read .las/.laz, detect and reproject CRS to UTM, apply statistical outlier removal, normalize elevation, save ground-ready point arrays.
laspyPDALOpen3DInput: .lazOutput: clean_points.npy
M2
AI Point Cloud Classification
Deep learning semantic segmentation to label each point: ground, building, vegetation, road, water. Replace rule-based CSF with RandLA-Net for large-scale outdoor LiDAR. Target: mIoU > 75%, Ground F1 > 90%.
RandLA-NetPyTorchKPConv (alt)Input: clean_points.npyOutput: classified_points.npy
M3
DTM Generation
Extract ground class points, apply TIN-based Delaunay interpolation, convert to 1m raster, export GeoTIFF with correct CRS. Validate RMSE against ground control points.
GDALRasterioscipy.griddataInput: ground_points.npyOutput: dtm.tif
M4
Hydrological Modeling Engine
Fill sinks (Planchon-Darboux), compute D∞ flow direction, derive flow accumulation raster, threshold to natural streams, delineate micro-watersheds, identify depressions and low-lying zones.
WhiteboxToolsTauDEMRichDEMInput: dtm.tifOutput: flow_acc.tif, streams.shp
M5
Flood & Waterlogging Prediction
Per-cell feature extraction (elevation, slope, curvature, flow accumulation, distance to road, impervious ratio, depression depth). Train XGBoost classifier on labeled flood records. Output probability raster; threshold at 0.7 for hotspot polygon.
XGBoostLightGBMscikit-learnInput: dtm.tif + flow layersOutput: flood_risk.tif, hotspots.shp
M6
Drainage Network Optimization
Build terrain graph: nodes = flood hotspots + stream junctions + road crossings; edges = feasible drainage paths. Cost function: distance + λ1·elevation_gain + λ2·built_area_penalty + λ3·excavation_volume. Solve with A* (hackathon) → NSGA-II (production). Output polyline shapefile + culvert points + excavation volume estimate.
NetworkXDEAP (NSGA-II)NumPyInput: hotspots.shp + dtm.tifOutput: drainage_network.shp
M7
GIS Output & Dashboard
Serve COG GeoTIFF rasters via GeoServer, package vector layers as shapefiles. Web dashboard with toggleable DTM, flow, flood, and drainage layers. Climate simulation slider (50–200mm rainfall). Before/after drainage comparison view.
FastAPIPostGISGeoServerReact + LeafletOutput: dashboard + GIS bundle

Section 5 — Differentiating Innovations
What separates this from every other team
Climate Resilience Simulation
Slider for 50/100/200mm rainfall events. Shows drainage overload zones, failure points, and resilience score. Makes the system forward-looking, not just descriptive.
Explainable Flood AI
Click any flood hotspot to see SHAP-driven explanation: why this zone floods, contributing upstream catchment map, AI confidence score. Builds planner trust.
Village Drainage Risk Index
Composite score = w1·(low elevation) + w2·(poor drainage density) + w3·(impervious ratio). Single number for government prioritization across villages.
Cost-Aware Planning Mode
Auto-generates estimated excavation volume, budget approximation, and phase-wise implementation priority. Bridges data analysis and government budget cycles.
Before vs After Simulation
Interactive map comparison: flood extent without drainage vs with proposed network. Visual proof of impact for judges and policymakers.
Multi-Village Meta-Learning
Train on SVAMITVA data from Gujarat, fine-tune for Tamil Nadu via domain adaptation. Demonstrates scalability and transfer beyond demo village.

Each person depends only on their input file, never on another person's running code. All files shared via Google Drive under /data/, /code/, /outputs/. No blocking dependencies.

Section 8 — Risks & Mitigations
Known failure points and fallbacks
Low classification mIoU
Ensemble RandLA-Net + CSF filter; fine-tune on village-specific labels. Fallback: percentile height threshold for ground.
Flat terrain / flow errors
Switch from D8 to MFD for multi-directional flow distribution. Increase sink-fill depth tolerance.
GPU memory overflow
Apply voxel downsampling before inference. Tile point cloud into 256m × 256m blocks.
No flood ground truth labels
Use synthetic labels from depression depth + flow accumulation thresholds. Document clearly as proxy labels.
Optimization runtime too slow
Use A* (fast, explainable) for demo. Pitch NSGA-II as production upgrade. Pre-compute graph edges.
Time shortage
M5 (Flood ML) and M6 (Dashboard) are drop-safe. Core demo = M1–M4 + M6 (basic map). Never drop M7 (GIS export).

Section 10 — Scalability & Deployment
From one village demo to national infrastructure
Tile-based processing
Divide village into 256×256m blocks, process in parallel using Dask or Ray. Linear scaling with added compute nodes.
Cloud storage (COG)
Rasters stored as Cloud-Optimized GeoTIFF on S3 or NIC MeghRaj. Supports on-demand tiled serving with no full-file download.
Government integration
REST API for Panchayat GIS systems, PMGSY, and Jal Jeevan Mission. Hosted on NIC MeghRaj for data sovereignty.
Multi-state transfer
Train on Gujarat SVAMITVA data, fine-tune for Tamil Nadu via domain adaptation. Meta-model learns terrain patterns across regions.
